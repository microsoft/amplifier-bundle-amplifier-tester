# Evaluation 01: Setup with build-up-foundation

A worked example of the amplifier-tester `setup-digital-twin` agent handling a
realistic, slightly-imprecise user ask. Single-variant capability eval — does
the agent correctly resolve the ask and produce a working DTU?

The evaluation runs entirely inside an outer DTU (DTU-in-DTU), so the test
never touches the host's Incus/Docker/Gitea state.

## Target of evaluation

The `amplifier-tester:setup-digital-twin` agent. Specifically, its ability to
take a natural-language ask that does not enumerate exact repo paths, infer
the intended bundle and CLI install, generate a working DTU profile, launch
the DTU, and hand back interactive entry commands.

## Hypothesis

A user asking "set up a clean DTU for running amplifier-app-cli with the
build-up-foundation bundle from amplifier-foundation experiments dir" wants:

- a fresh DTU
- amplifier CLI installed
- `build-up-foundation` composed as an `--app` bundle
- a hand-back report with the four pieces needed to use the DTU

The agent should resolve `build-up-foundation from amplifier-foundation
experiments dir` to the bundle entrypoint at
`experiments/build-up/build-up-foundation.md` in amplifier-foundation and
produce a profile that installs amplifier + composes that bundle from
`git+https://github.com/microsoft/amplifier-foundation@main#subdirectory=experiments/build-up/build-up-foundation.md`.

## Scenario (verbatim prompt to the agent)

See [`scenario.md`](./scenario.md). The prompt is intentionally imprecise —
testing the agent's resolution is part of the point.

## Setup (DTU-in-DTU)

```
Outer DTU profile:  profiles/outer-dtu.yaml
Outer DTU contains: Incus + Docker + Amplifier CLI + amplifier-digital-twin
                    + amplifier-gitea + amplifier-tester bundle (from GitHub @main)
Inner DTU:          generated by the agent at runtime, inside the outer DTU
Sample count:       1 run per invocation
```

The outer DTU is a fork of `amplifier-bundle-digital-twin-universe`'s
`dtu-host-in-incus.yaml` profile, with the `amplifier-tester` bundle composed
(transitively brings the DTU and Gitea bundles).

## Success criteria

Scored by `criteria/check.sh`:

1. **profile-generated** — agent wrote a profile YAML somewhere readable in
   the outer DTU (typically `/tmp/amplifier-tester/profile-*.yaml` or
   `/root/.amplifier/digital-twin-universe/profiles/`).
2. **inner-dtu-running** — `amplifier-digital-twin list` inside the outer
   DTU reports the inner DTU in status `running`.
3. **amplifier-version** — inside the inner DTU, `amplifier --version`
   exits 0.
4. **build-up-foundation-bundle** — inside the inner DTU,
   `amplifier bundle list` shows `build-up-foundation` registered.
5. **inner-smoke-run** — inside the inner DTU,
   `amplifier run "Say exactly: build-up-foundation-ok"` returns the exact
   string (stresses the bundle's "parent has no tools, must delegate"
   structural constraint).
6. **handback-complete** — the agent's stdout contains all four hand-back
   pieces: instance ID, profile path, an interactive
   `amplifier-digital-twin exec <id>` command, and a destroy command.

## How to run

```
./run.sh
```

Runner steps:

1. Preflight — verify `amplifier-digital-twin` CLI, `incus`, `ANTHROPIC_API_KEY`.
2. Launch the outer DTU from `profiles/outer-dtu.yaml`.
3. Push `scenario.md` into the outer DTU at `/root/scenario.md`.
4. Run an Amplifier session inside the outer DTU, passing the scenario as the
   prompt. Capture stdout/stderr and the session directory.
5. Run `criteria/check.sh` against the resulting inner DTU.
6. Write `meta.json`, render `verdict.md` and `verdict.html` summarizing
   pass/fail.

The runner is idempotent — re-running on the same day creates `run-2`,
`run-3`, etc. Each run writes to `results/<YYYY-MM-DD>/run-N/`.

The outer DTU is **left running by default** so you can inspect state.
Destroy it manually with the command printed at the end of `run.sh`.

## How to read results

```
results/<date>/run-N/
  meta.json              # outer/inner DTU IDs, session ID, timestamps
  stdout.txt             # full agent output (the user-visible response)
  stderr.txt
  outer-dtu-launch.json  # JSON from outer DTU launch
  inner-dtu-list.json    # `amplifier-digital-twin list` inside outer DTU
  criteria.txt           # PASS/FAIL per criterion + summary line
  metrics.json           # structured pass/fail + run metadata
  verdict.md             # rendered markdown summary — start here
  verdict.html           # self-contained HTML report — share/email this
  sessions/              # pulled session directory from outer DTU (if any)
```

`python3 metrics/extract_metrics.py results/<date>/run-N/` re-extracts metrics
from a captured run.

## Limitations / known trade-offs

- The eval installs `amplifier-tester` from GitHub `@main`, not from the
  local workspace. To test local (unpushed) amplifier-tester changes, mirror
  via Gitea — see `amplifier-tester-self-validation.yaml` for the pattern.
- Single sample per run. The setup-digital-twin agent is non-deterministic;
  re-run for variance.
- Total wall time is ~15-30 minutes per run (outer DTU provisioning + agent
  reasoning + inner DTU provisioning + smoke runs).
